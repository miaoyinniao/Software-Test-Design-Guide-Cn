介绍

对于脚本测试的理解，它必须了解其历史背景。脚本测试作为瀑布软件开发模型的一个组成部分出现。瀑布模型定义一系列的开发阶段，每个阶段有出入口条件，必做活动，交付件（有形的工作成果）。这是“做出计划，然后按计划行事”哲学的典型例子。典型的瀑布阶段包括：

  1. 系统需求-需求收集

  2. 软件需求-系统软件部分的需求收集

  3. 需求分析-分析，分类，重定义软件需求

  4. 程序设计-定义系统的架构，模块，接口选择

  5. 代码编写-写代码实现设计

  6. 测试-评估需求是否被正确理解（Validation）设计是否通过代码正确实现 (Verification)

  7. 运营- 系统产品化

奇怪的历史说明

现在Winston Royce被公认为软件开发瀑布模型之父。实际上，在他的论文中提出了早期模型迭代和增量开发的概念-一些现今很多公司正在运用的。


     现在我们对于脚本测试有不同的看法，任何开发方法包括从瀑布到快速软件开发都需要使用脚本测试，每当重复性，审计性，客观性显得重要时，就会用到脚本测试。
重复性是指非作者用相同的方式能够执行的一定细节程度的测试的定义（从设计带详细定义）。
客观性是指测试创建不依赖于人员技能（就像魔法）但是基于典型的测试设计原则。审计性是指从需求，设计，编码，测试用例能够回溯和跟踪。这样才能有效评估测试覆盖。


"Plan your work, work your plan."没有更好的短语能够解释脚本测试方法了，具体文档参见 IEEE Std 829-1998, the "IEEE Standard for Software Test Documentation."

该标准定义了8个文档用于定义软件测试，如下：

  • Test plan

  • Test design specification

  • Test case specification

  • Test procedure specification

  • Test item transmittal report

  • Test log

  • Test incident report

  • Test summary report

Figure 12-2 展现了这些文档的关系。注意前面4个文档定义的test plan, test designs, and test cases 都是在产品开发前定义的，实际上测试已经开始。脚本测试关键点是-测试计划基于正式的系统需求。


Figure 12-2: The IEEE 829 Test Documents

奇怪的是, the IEEE 829 standard描述, "该标准规范了测试文档的格式和内容，但没有要求必须写这些文档“换句话说，你可以不写这些文档，因此测试人员和组织可以有选择的进行。但是，标准要求一旦你写测试计划等文档，就需要遵循标准。

The IEEE 829 standard 列出了以下优势:

  • 一个标准化的测试文件可以方便沟通，提供一个共同的参照框架。

  • 一个标准化的测试文档的内容定义，可以作为相关测试过程的完整清单。

  • 可以提供一套标准化的基线测试文档，以便用于评估。

  • 这些文件的使用大大增加了测试的可管理性。可管理性增加的结果是大大提高了测试过程中的每个阶段的可视化。
