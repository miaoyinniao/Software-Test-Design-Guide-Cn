12.1 IEEE 829文档描述

IEEE 829文档描述

The IEEE 829 标准定义8个不同的文档，每份文档由多个章节组成。

测试计划

  • 测试计划的目的是描述范围，方法，资源，测试活动计划，它包含需要测试的项目（组件）和特性（功能，性能，安全，可用性等），需要完成的任务，交付件（有形工作成果），测试职责，时间表，标准要求。测试计划可以是总体计划（主测试计划）或者子级别的计划（单元，集成，系统，验收等）。它由以下章节构成。

    1. 测试计划ID-区别于其他文档的唯一ID。

    2. 介绍 - 被测软件的总括描述，包含简要描述历史和上下文，可参考的背景资料（对于理解被测试软件有帮助）.定义陌生的术语。

    3. 测试项 - 标示出需要测试的软件项（item），item是故意含糊的，它是被测对象的”块“ 。

    4. 需要测试的特性-标示出测试项的特性，包含功能，性能，安全，可移植性，可用性等。

    5. 不需要测试的特性-标示出不需要测试项的属性和原因。

    6. 方法-整体的测试方法，以确保所有的项目和其功能将得到充分的测试。

    7. 测试项通过/失败标准-标准用于确定是否每个测试项已经通过测试。

    8. 暂停测试和恢复测试标准-暂停测试的条件和恢复测试的条件。

    9. 测试交付件-标示出测试过程中需要交付的文档和输出。

    10. 测试任务-需要完成的测试活动。

    11. 环境需求-测试活动需要的特定环境：硬件，软件，网络，设施，工具，人员等。

    12. 测试职责-标示出执行测试任务的人员/群组的职责。

    13. 人事和培训需求-制定人员数量和人员要求，包括技能等。

    14. 时间表-定义测试过程中的关键里程碑事件。

    15. 风险和应急措施-识别测试计划中的高风险，确定预防和应急/补救措施。

    16. 评审 - 确定评审该文档的人员姓名和角色。

测试设计说明书

  • 测试设计说明书的目的是给出待测试的特性集合和描述充分测试这些测试特性的测试用例。另外，需要描述测试计划中给出的测试方法的细化。测试设计说明书由下列章节组成：

    1. 测试设计说明书ID-区别于其他文档的唯一ID

    2. 需要测试的特性-标示出测试项和特性

    3. 测试方法的细化-确定需要用到的测试技术。

    4. 测试识别-列出测试设计相关联的测试用例，提供唯一ID和每个用例的简要描述。

    5. 测试特性通过/不通过标准-标准用语判决测试特性是否通过。

测试用例说明书

  • 测试用例说明书的目的是确定和细化测试设计说明书列出的用例。测试用例说明书由以下部分组成：

    1. 测试用例说明书ID-区别于其他文档的唯一ID

    2. 测试项-标示测试项和特性。

    3. 输入-确定测试用例的输入。

    4. 输出-确定测试用例的预期输出。

    5. 环境需求-特定的硬件，软件，设施等，特别是在测试计划中没有列出来的。

    6. 特定的过程要求-定义测试用例需要的安装，执行，清理等过程需求。

    7. 内部依赖-列出测试用例执行优先级。

测试过程说明书

  • 测试过程说明书的目的是确定测试用例的执行步骤和判决软件是否通过测试的过程。测试过程说明书由以下部分组成。

    1. 测试过程说明书ID-区别于其他文档的唯一ID。

    2. 目的-描述测试过程的目的和相对应的测试用例。

    3. 特殊需求-列出测试执行过程的特殊需求。

    4. 过程步骤-列出过程步骤，也许包括： Set up, Start, Proceed, Measure, Shut Down, Restart, Stop, and Wrap Up.

转测试报告 (又名Release Notes)

  • 转测试报告的目的是确定已经提供的可测试项，它有以下部分组成：

    1. 转测试报告ID-区别于其他文档的唯一ID。

    2. 转测试项-列出高版本或者分支的转测试项。

    3. 地点-标示出转测试项的地点。

    4. 状态-描述转测试项的状态，包含需求的变更等。

    5. 评审- 确定转测试评审人和角色。

测试日志

  • 测试日志的目的是提供按时间顺序排列的记录，包含测试执行中的相关细节。它由以下部分组成：

    1. 测试日志ID-区别于其他文档的唯一ID.

    2. 描述-标示测试项和测试执行环境。

    3. 活动和事件输入-对于每个事件，列出开始和结束日期和时间，测试执行，测试结果，环境信息，异常事件信息，缺陷报告ID等。

测试缺陷报告(又名Bug Report)

  • 测试缺陷报告的目的是记录在测试过程中观察到的任何事件以便需要进一步调查，它由以下部分组成：

    1. 测试缺陷报告ID- 区别于其他文档的唯一ID。

    2. 简述-简要描述缺陷。

    3. 详细说明-说明测试输入，预期结果，实际结果，测试环境，重现条件等。

    4. 影响-描述缺陷对其他测试项的影响，并说明进一步的测试建议，如果有必要的话。

测试总结报告

  • 测试总结报告的目的是总结测试活动和测试结果，提供基于结果的评估。它由以下部分组成：

    1. 测试总结报告ID-区别于其他文档的唯一ID。

    2. 概述-概述测试项的评估结果。

    3. 偏差- 报告和预期结果的偏差。

    4. 综合评价-结合测试计划中的标准全面评估测试过程。

    5. 测试结果总结-总结测试结果，标示出所有遗留缺陷。

    6. 评价-给出所有测试项的评价包含测试项的限制条件。

    7. 测试活动总结- 总结重点测试活动中的任务和资源使用。

    8. 评审-确定转测试评审人和角色
