状态转换图
从非正式定义的角度解释状态转换图很简单，既然Brown & Donaldson和Stateless University Registration System包含太多基于需求的转态转换，我们举另外的例子，为了取得非州立大学资格，我们需要空中预定系统，让我们呼叫最喜爱的公司（Grace L. Ferguson Airline & Storm Door Company）预定，我们提供信息包括出发地，目的地，时间，次数，一个预订系统代理，提供接口将这些信息送给航空公司的预订系统。这种情况下，Reservation处于Made状态，另外，系统创建和启动了一个定时器，每个系统预定有规则保证什么时候必须付款。这些规则基于目的地，服务类型，日期等信息，如果定时器到期了，那么预定会取消，在状态转换标记这些信息如下：
 
Figure 7-1: The Reservation is Made.

圆圈代表一个状态，这个例子中，Reservation处于Made状态，箭头显示了到Made状态的转换，箭头上的描述“giveInfo”，是指从外部世界到系统的事件。命令/表示动作“startPayTimer”，黑点表示图的开始。
在Reservation已经是Made状态，但是在PayTimer超时之前已经付款，Reservation就进入Paid状态，增加的箭头“PayMoney”。
 
Figure 7-2: The Reservation transitions to the Paid state.

在继续之前我们定义关系更正式化：
  • 状态（圆圈）-一个状态需要在系统等待一个或多个事件的条件。状态记录了系统在过去的输入和定义系统在接下来事件发生时的回应。这些事件可能导致系统状态转换和/或回到初始态。状态一般代表系统中一个或多个变量的值。
  • 转换（箭头）-代表一个状态因为事件导致的到另一个状态的抓换。
  • 事件（代表转换中标记）-一个事件是导致系统发生状态变化的东东，一般认为，是指外部世界的事件进入系统接口，有时候伴随系统比如Timer expires 或者Quantity on Hand goes below Reorder Point。事件是瞬时的，事件可以是独立的或者相互关联的（事件B不能在事件A之前发生）。当事件发生时，系统能够转换状态或者保留同样状态和/或执行一个动作。事件也许和参数相关联，比如，Pay Money 也许表示Cash, Check, Debit Card, or Credit Card.
  • 动作（命令在/之后）- 状态转换的操作开始，他可以是print a Ticket, display a Screen, turn on a Motor等，这些动作导致一些东东创建并且系统输出。注意动作发生在状态转换之间，状态自身是被动的。
  • 状态转换图的输入点是一个黑点，总结点是一个眼睛点。
这些标记是Mealy发明的，一个替代的标记法是Moore定义的但很少使用，更多信息参考 Fowler and Scott的书《UML Distilled: A Brief Guide To The Standard Object Modeling Language》，它详细讲解了更复杂的问题，比如分组和嵌套状态转换图。
注意状态转换图代表一个特定的实体（Reservation），他描述预定的状态，事件影响状态，预定的转换从一个状态到另一个状态，动作是初始化于预定。一个普遍的错误是混淆不同实体到一个状态转换图中，例子中容易混淆Reservation和Passenger。

完成可吗？不是，客户优势取消预订，从Made状态客户要求取消预定，一个新的状态，Cancelled By Customer

另外，一个预定可以从Paid取消，这种情况下，退款需要添加，然后退出系统，状态还是Cancelled By Customer.

最后一个添加，从Ticketed客户取消预定，这种情况下退款需要添加，下个状态是 Cancelled by Customer，但是这个是有条件的，航空公司需要拿到机票才能退款，一个新的标记[]出现了，是指条件下转换。

 
Figure 7-9: Cancellation from the Ticketed state.

注意图仍然不完整，没有箭头可以把Cancelled状态转换到终点，也许我们可以从Cancelled NonPay状态恢复预定，我们继续宽展到座位选取，航班取消，其他重要事件，但这些已经足够说明该技术了。
就像描述的一样，状态转换图将复杂的系统规则和交互简化表达出来了，非常希望的是，当复杂性存在时，分析师和设计师能用状态转换图表达需求以帮助他们设计。

创建测试用例
在状态图中的信息，易于创建测试用例，四个不同层级的覆盖定义如下：
1，创建用例覆盖所有状态至少一次，3个用例可以覆盖，这个是最薄弱的覆盖。
 
Figure 7-10: A set of test cases that "visit" each state.
2， 创建
例覆盖所有事件至少一次，3个用例，也是弱覆盖。
 
Figure 7-11: A set of test cases that trigger all events at least once.
3，创建用例覆盖所有路径至少一次，虽然测试层级是优先的因为覆盖层级，但也许不太可行，如果状态转换是循环的，比如A到B，B到A，可能路径如下：

A→B

A→B→A

A→B→A→B→A→B

A→B→A→B→A→B→A

A→B→A→B→A→B→A→B→A→B
还有更多，测试循环是很有必要的，因为累积之后计算错误或者形成资源耗尽导致问题发生，比如锁处理不匹配，内存泄露等。
关键点： 测试所有转换往往是我们推荐的测试层次。
4，创建用例覆盖所有转换至少一次，该层次测试提供很好的层级覆盖，并且不需要生成大量用例， 
Figure 7-12: A set of test cases that trigger all transitions at least once.
